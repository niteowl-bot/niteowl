## 2026-07-20 (Calendar & Appointment Management — Step 2: read real-time availability)

### Added — additive; reuses the existing booking engine, no new booking system
Second step of the phased Google Calendar integration. When a business has connected its calendar (Step 1), Remy now checks the calendar's **real-time free/busy** before offering or confirming a slot — so a time already blocked on the business's own Google Calendar (an event booked outside Remy) is never offered. Layered onto the **one** shared availability function every surface already funnels through, so chat, widget, and voice-after-call all gain this with **zero per-surface changes and no change to `leadCapture.ts`**.
- **`src/lib/calendar/freebusy.ts`** (new): `getCalendarBusy(orgId, start, end)` reads the connected calendar's busy intervals via Google's `freeBusy` REST endpoint, using Step 1's authenticated connection + token refresh. Returns a three-way result — `no_connection` (calendar plays no part), `ok` (authoritative busy list), or `error` (connected but unreadable). Plus a pure `slotConflictsWithBusy()` overlap test (half-open intervals).
- **`src/lib/availability.ts`** (the only existing file changed): `isSlotAvailable` and `findNextAvailableSlot` now apply a second, additive layer after the existing business-hours + internal capacity checks — the slot must also be free on the connected calendar. Key safety properties:
  - **Busy slots are never offered**: a calendar conflict makes the slot unavailable, and the suggested alternative is drawn only from calendar-verified slots.
  - **Fails closed on calendar error**: if a calendar is connected but can't be read (expired/revoked token, API down), the slot is treated as unavailable and no alternative is suggested — Remy never confirms a time it couldn't verify, prioritising "never double-book" over convenience. (Orgs use the existing capture/awaiting-confirmation path, so the lead is still captured; nothing is auto-confirmed.)
  - **Unconnected orgs are completely unaffected**: `no_connection` short-circuits to the pre-existing capacity result — behaviour is byte-for-byte the same as before for every business that hasn't connected a calendar.
  - **One free/busy call per search**: `findNextAvailableSlot` fetches the calendar's busy list once for the whole search window and reuses it across candidate slots (passed via a new optional `opts` on `isSlotAvailable`), rather than one API call per candidate.
- **Buffer time** (`docs/sql/2026-07-20_booking_buffer_minutes.sql` + verify): new additive `organisations.booking_buffer_minutes` (default 0). When set, it widens the window a slot is tested against calendar events on both sides, enforcing a gap (travel time, cleanup) between a new appointment and existing calendar events. Default 0 = today's behaviour. Read in its **own** absence-tolerant query so the code is safe to deploy before the SQL runs (a missing column reads as 0 and never fails the critical capacity/settings queries, which deliberately don't select it). No UI to configure it yet (per this step's "don't redesign UI" scope) — DB-configurable for now; a later step surfaces it on the Calendar settings page.
- **Scope honoured**: reads availability only. Does **not** create/update/delete calendar events (Step 3/4), does not touch reschedule/cancel logic, voice behaviour, onboarding, pricing, or Outlook. The reschedule *availability check* in `bookings/manage` becomes calendar-aware automatically (it calls the same shared functions) — a safe, intended consequence that writes no events and changes no reschedule logic.
- **Verified** (2026-07-20): 17/17 unit assertions on the real `slotConflictsWithBusy` (overlap correctness + buffer-window semantics); 7/7 against the **real `isSlotAvailable`** on the live dev DB via injected free/busy states (no_connection→capacity result, ok+free→available, ok+busy→never offered, buffer enforces the gap, error→fail-closed); real `findNextAvailableSlot` still returns a valid in-hours slot for an unconnected org (no regression). Crucially, the real no-opts call returns unchanged results on dev even though **neither** the new column **nor** the `calendar_connections` table exists there yet — proving deploy-before-SQL safety. `tsc --noEmit` + `next build` clean. Live end-to-end against a real Google Calendar with known events is owner-gated on the Step 1/2 SQL + a connected calendar.

## 2026-07-20 (Calendar & Appointment Management — Step 1: connection foundation)

### Added — additive only, zero change to booking/availability/AI behaviour
First step of the phased Google Calendar integration (per an approved architecture plan). Lets a business securely **connect and disconnect its own Google Calendar** from Settings and see connection status. Deliberately inert with respect to bookings — reading availability from the calendar and writing/updating/cancelling real calendar events are later steps (2–4), so deploying this changes nothing a customer or the booking engine sees.
- **Schema**: new `calendar_connections` table (`docs/sql/2026-07-20_calendar_connections.sql` + `_verify.sql`), one row per org, mirroring the `voice_settings` pattern exactly — `org_id` PK, service-role-only writes, owner read-only RLS. `provider` carries a check constraint including `'outlook'` so a second provider needs no schema change. Not yet run on dev/prod; hand-run via the Supabase SQL editor per the repo's no-migrations convention.
- **Provider abstraction** (`src/lib/calendar/`): `provider.ts` (`CalendarProvider` interface) + `google.ts` (Google impl via the new `google-auth-library` dependency; OAuth lifecycle + userinfo, Calendar REST left as raw `fetch` for later steps) + `connection.ts` (the token store: load/save/refresh/disconnect). Modelled on billing's provider/impl/agnostic split so Outlook slots in later with one line and no caller changes.
- **Token encryption at rest** (`src/lib/calendar/tokenCrypto.ts`): OAuth access/refresh tokens are encrypted with AES-256-GCM (new `CALENDAR_TOKEN_ENC_KEY` env var) before touching the database and decrypted only server-side — refresh tokens are the one genuinely long-lived credential this app stores. Round-trip, tamper-detection, and wrong-key rejection all verified.
- **OAuth routes** (`src/app/api/calendar/google/{connect,callback,disconnect}`): owner-auth'd connect issues the Google consent redirect (offline access, `prompt=consent` to guarantee a refresh token, minimal `calendar.events` + `calendar.readonly` + `openid email` scopes) with a random CSRF state stored in an httpOnly cookie; callback validates the state, re-resolves the org from the session (never trusts the redirect), exchanges the code, and stores the encrypted connection; disconnect revokes at Google (best-effort) and clears the row.
- **UI**: `Settings → Calendar` page (new nav tab, additive) with Connect / "Connected as `<email>`" / Disconnect, reusing the existing design system.
- **Setup runbook**: `docs/CALENDAR_SETUP_RUNBOOK.md` covers the owner's Google Cloud setup, using Google's **test-users** path to skip the weeks-long sensitive-scope verification during alpha (capped at 100 users).
- **Verified from here** (without real Google credentials, 2026-07-20): token crypto (round-trip/tamper/wrong-key); `tsc --noEmit` + `next build` clean (`google-auth-library` bundles under Turbopack with no `serverExternalPackages` entry); unauth `/connect` → `/login`; authed `/connect` produces a correct Google consent URL + httpOnly state cookie; callback rejects a missing/mismatched state; the Settings page renders "Not connected" and degrades gracefully even before the SQL table exists. **Not touched** (confirmed by diff): `leadCapture.ts`, `availability.ts`, `bookings/manage`, chat/widget/voice, onboarding, pricing, or any existing table. Only pre-existing files modified: `package.json` (+dependency) and one additive line in the Settings nav.
- **Owner-gated remainder**: the real end-to-end OAuth round-trip needs the dev/prod SQL run, a Google Cloud project + test user, and the three env vars set — tracked in CHECKLIST.md.

Widened both `select()` calls to the full column list; `handleUpdate`'s `setRecords` now merges (`{...r, ...data}`) instead of replacing, so this class of gap can't recur if another field is added later without every caller remembering to re-select it.
- **Bug found and fixed**: the Publish/Unpublish button had no in-flight guard, so a second click before the first request's response landed would read a stale `record.status` and immediately toggle the value back — a real, successful write looked like "nothing happened." Same root cause on the Merge/Replace/Keep-both banner buttons. Both now disable themselves and show explicit "…"/"Saving…" feedback while a request is in flight.
- **Bug found and fixed**: `findLikelyDuplicate`'s word-matching had no stemming — "cleaning"/"cleaner" vs. "clean" and "hour" vs. "hours" didn't match as strings, silently missing an obvious duplicate ("Standard Cleaning Service" vs. existing "Standard Clean" scored 0.5, under threshold). Added lightweight suffix stripping (-ing/-ers/-er/-es/-s). Re-verified the previously-passing "must NOT match" cases still don't.
- **Bug found and fixed**: `PATCH /api/knowledge/import/items/[itemId]`'s "duplicate must be resolved before approving" check only looked at whether *that specific request's body* included `duplicate_action` — but Merge/Replace/Keep-both is saved via its own earlier, separate PATCH (clicking the banner button). Approving sends a follow-up PATCH containing only `{review_status: "approved"}`, so the check saw no `duplicate_action` in *that* request and blocked an already-resolved item. Now checks the item's existing database value as a fallback (`raw.duplicate_action ?? item.duplicate_action`).
- **Gap found and fixed (not a bug — a design hole)**: duplicate detection only ever checked a new item against the *already-committed* Knowledge Base, never against other items extracted in the *same* import batch. A single document naturally produces both a structured KB entry ("Call-out fee", `price: 10, currency: GBP`) and an AI-generated FAQ restating the same fact ("Do you charge a call-out fee? Yes, £10...") — reviewed side by side with no relationship between them. Editing one later (confirmed live: a reviewer changed the FAQ's £→€ without knowing the sibling entry existed) leaves the two silently disagreeing, and Remy answers from whichever one it happens to use. Added `findLikelyDuplicateAmong()` — checks each new item against everything staged earlier in the same batch and skips staging the redundant one entirely. Required generalizing the match scorer to check both containment directions (short entry fully covered by a longer paraphrase, and vice versa) and keep the better score — the one-directional version missed this exact FAQ/KB-entry pair (0.5 one way, 1.0 the other). Re-verified against the full existing regression suite plus a short-vs-long sanity case; all pass. Manually reconciled the two live entries this was found on (`Call-out fee` / `Do you charge a call-out fee?`) to both read €10.

## 2026-07-16 (Owner notification recipient — separate from login email)

### Changed (`src/lib/leadCapture.ts`, `.env.local`, new `docs/sql/2026-07-16_organisations_notification_email.sql`)

## 2026-07-06 (AI-call reliability bundle)

### Fixed
- **No OpenAI call anywhere had a timeout.** A hang or slow response from OpenAI (routine, not hypothetical) would previously leave a request running until the platform's own function timeout killed it, with no fallback message — directly threatening the "never miss a customer enquiry" promise. - **Dashboard preview chat's `streamChat()` (`src/lib/chat.ts`) had no error handling at all**, unlike `public/widget.js` which already wrapped its fetch/stream loop in `try/catch/finally`.
